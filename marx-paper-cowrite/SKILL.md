# 马理论科论文写作（引导式·和用户共写）

## 这个 skill 解决什么问题

quickwrite 一气写整篇，快；但用户思想不参与，写出来对用户是陌生的，改起来压力大，也容易跟用户本意跑偏。cowrite 是引导式，把写作拆成**一段一段**写，每段写之前先问用户、收想法，再结合用户的"阅读思考"落笔写这段，写完过目、随时改。慢，但用户思想全程参与，文章更有灵魂、更贴合用户本意。

两种写法是并列二选一，不是先后关系。赶时间、不想多介入的用 quickwrite；要让用户思想参与、想边写边和用户聊的用 cowrite。

**位置**：和 quickwrite 一样，在 marx-paper-structure 结构定稿 + marx-paper-research 素材整理之后运行。写完衔接 marx-paper-review。

## 什么时候选 cowrite

用户明确说要引导式/共写式/"和我一起写"/"你问问题我来想"/"边读边写"/"一段一段写"时选本 skill；赶时间或不想多介入用 quickwrite。

- 若前面跑过 marx-paper-read：工作区有"阅读思考/用户想法/"，用户读报告时冒的想法已经落盘——这是本 skill 最重要的输入，问问题就围绕这些展开
- 若没跑过 marx-paper-read：本 agent 自己读那些参考文献、自行想几个有启发性的问题问用户——但这只补一刀，仍然以先问用户、收用户想法为先

## 核心机制：提问启发 + 一段一段写

流程按结构方案的段落顺序推进，一段一段写，从引言到主体 3×3 的 9 个部分再到结语。**每写一段都走下面四步循环**：

1. **回归当前段落的定位**：读结构方案，明确这段是哪个部分、它要承担什么论证任务、字数多少
2. **提问启发**：写这段前，先问用户**几个有启发性的问题**，激发用户对这段的观点、判断、能想到的例子。问题要有质量：
   - 问得**具体**指向这段要解决的那个分论点，问"空泛"的问题（如"你怎么看自由"）问不出东西
   - 问得**有启发性**：从用户之前的阅读思考里提炼出张力、空白、矛盾点，问得让用户不得不表态或联想。可以从"用户想法/"里挑一条没解决的追问
   - 问得**结合用户本意**：用户在前面思考里对这有什么想法就往那个方向问，让用户感觉在被认真接住，不是被考
   - 一次问 2-3 个问题，别问一大串堵住用户；问完等用户答，不要自问自答
3. **收用户想法、结合素材落笔写这段**：拿用户刚答的想法 + "阅读思考/用户想法/"里对这段有用的 + 素材库里本次能用的引用/史实/数据，按 quickwrite 的写作规范写出这段。这一段是"边读边写"的产出，要把用户思想揉进去、但写成马理论科学术语体（用户想法是原料，写成论文是 agent 的活）
4. **请用户过目、可随时改**：把这段给用户读，请他看有没有不对/不全/不喜欢的地方。用户提意见就改，改完再请他确认通过才进下一段。**通过一段才写下一段**，不积压

这就是"一段一段写"的本质：不是简单把整篇拆开写（那样没意义），而是**每段都先经过提问启发 → 收想法 → 落笔 → 用户过目改**这个循环。用户的思想和判断全程参与。

## 提问怎么问才有启发性（核心，别水)

cowrite 的价值全在提问这一步。问得不好就退化成"用户当枪手把想法说一遍"或"agent 自己写、用户只点头"。提问怎么做：

- **基于用户的阅读思考问**：先读"阅读思考/用户想法/"里跟这段相关的（用户之前读报告时冒的想法、保留意见、点过的可用之处）。这些是用户**已经表态过**的东西，绕着它们问就把用户的思想接上了。例如用户想法里写过"弗洛姆把自由说成逃避是太悲观了"——问这段时就可以挑这个点："你在读弗洛姆时觉得他太悲观，这段要谈他对自由的态度，你这个判断要不要在这段表达出来？怎么表达？"
- **没走过 read 怎么办**：agent 自己读那批参考文献里跟这段最贴的 1-2 篇、想几个问题问。但这是补救手段——最理想还是先走 read，让用户的思考先落到盘上
- **问张力、问立场、问具体证据**，不问"你怎么看这个话题"这种空题。例：
  - 张力题："A 学者说 X，B 学者说 Y，你这段要谈这个问题，你站哪边、为什么？"
  - 立场题："这段如果让你一句话说清你的判断，怎么说？"
  - 证据题："你说主张是这个，那你能想到用什么支撑？原文/史实/案例哪些能用？"
- **问得让用户表态**：用户在前面思考里模湖的地方，正是这段要让用户明确立场的地方，问进去
- **一次 2-3 个，等他答**：别连珠炮。用户答了再追问、再澄清，动态对话，不是一发问就出整段

## 框架可变原则

结构方案是 V1，不是死契。一段一段写的过程中，用户发现某部分其实该拆/该并/该挪顺序、或后面某部分有了新想法——随时可以改框架。agent 要主动提醒用户"要不要顺手调整结构方案"，必要时升一个版本号存盘（V2/V3 保历史版本，同 marx-paper-structure 的做法）。别把结构方案当成不可动的约束。

## 写作规范沿用 quickwrite

cowrite 写出的每一段，遵循的写作规范与 marx-paper-quickwrite 完全一致，本 skill 不另立一套。沿用那边的全部规则——

- 先读结构方案严格按字数写
- 不被研究资料的文体带偏（禁鲁迅腔/禁散文腔/禁格言式/禁口语化）
- 克制 AI 典型句式（"不是而是/不仅更是/并非而非"整篇三五处以内）
- 引用学者观点必须指名（红线：出现"一些学者""有学者认为"=硬伤）
- 每部分要有支撑但贵精不贵多，整部分无任何支撑=硬伤（别硬写，告诉用户去找材料）
- 克制破折号、克制反问句
- 用马理论科学术语体，避免时评/公文/网络的词
- 引号 = 逐字原文（硬规则）；原著用最新最全版本；溯源规则（人物原话溯源到原始出处，不标"转引自"）
- 间接引用换说法、每篇最多 1 条、指向二手论文核心论断、被引需高被引/知名学者、期刊著录从知网复制、同一文献多条脚注页码逐一核实
- 十种写作方法（开宗明义/逻辑与历史统一/经典互文/对立统一/批判-建构破立结合/论-据-证闭环/历史条件分析/物质利益揭示/概念界定/始于问题终于实践）

详细见 marx-paper-quickwrite/SKILL.md，本 skill 不重复全文，**两套写作规范同源**，避免一处改另一处忘了改的漂移。

## 工作流程

### 第一步：先理清要怎么写

开工前 agent 自己理一遍：
- 读结构方案（marx-paper-structure 定稿）：整篇分几段、各段定位、字数
- 读素材库（marx-paper-research 产出）：有哪些直接引用/间接引用/数据可用，大致对应哪些段
- 读"阅读思考/用户想法/"（若走过 read）：用户已经对哪些文献、哪些点表达过想法——这是提问的弹药
- 心中有"按什么顺序写、每段开问什么"的预案，但走起来随时调整

### 第二步：跟用户确认开写方式

跟用户对一下：一段一段写，从引言开始；每段先问你几个问题，你答完我结合你之前的阅读思考来写这段，写完给你过目、你说改就改、通过才写下一段。框架如果写到一半要调整随时说。让用户心里有底、知道接下来怎么配合。**不要一声不吭开始问问题**——先告知节奏。

### 第三步：按段落循环推进

从引言开始，每段走"四步循环"（回归定位 → 提问启发 → 收想法结合素材落笔 → 过目可改）。

- 写完一段、用户通过后，存盘：建议每段一份 `正文_第N部分_标题.md` 在"论文正文/"下，累积成篇。也可边写边攒成整篇，按用户习惯
- 用户在某段提"框架要改"：暂停推进，先调结构方案（升版本号存盘），再回到当前段
- 用户提"后面某部分我想到 XXX"：记到"阅读思考/用户想法/"对应那篇/那段下，写到那一段时再用，不立刻用
- 全程不死守"先引言后主体后结语"的顺序，**若用户想先写某部分也行**——但别擅自跳，由用户提

### 第四步：结语与通读

所有段落写完、逐段通过后：
- 写结语：和用户聊一下全文论点想收到什么收束效果，再落笔
- **通读整篇**：把各段拼成整篇全文通读一遍，看衔接（段与段之间是否顺畅、有没有重复或断裂），把衔接问题列给用户、和用户一起改。各段单独过目不能替代整篇通读
- 通读完把整篇落盘到"论文正文/"，定名如 `正文_V1.md`

### 第五步：衔接 review

写完进 marx-paper-review。本 skill 的逐段过目不能替代 review 的独立审核——review 是另一个独立 agent、用整套清单系统查，本 skill 的逐段只是"用户主观过目"，两者不同，都要走。

## 与 quickwrite 的取舍边界

- **赶时间、用户不想多介入** → quickwrite
- **要让用户思想参与、占用更多对话** → cowrite
- **课程论文、短篇**：可走 quickwrite；用户想慢慢磨、想真正吸收理解的，cowrite 也行
- **学位论文、要打磨**：建议 cowrite 或 cowrite 部分段落（关键段用 cowrite 磨、其余用 quickwrite 补），两者可混用——不必非整篇一种写法

混用时注意：同篇若一段用 cowrite 一段用 quickwrite，落盘到同一份 `正文_V*.md`，并在"论文正文/"下另存一份说明哪些段用了哪种、用户脑子里的想法记录见哪条。

## 衔接

- **上游**：marx-paper-research 结尾提醒用户选写法时被选到，或用户明确要引导式时路由到本。理想链路是 marx-paper-read（可选，用户思想先落盘）→ structure → research → 本 skill
- **下游**：写完进 marx-paper-review 独立审核，再进 marx-paper-format 排版
- **跟 quickwrite**：并列二选一，可混用

## 备注

- 当前是第一版，按用户设计"每段先问启发、再收想法结合素材落笔、过目可改、框架可变"实现。后续按实际使用磨合完善（尤其提问启发的"问法"细节、框架改动反馈机制）
- 提问启发是本 skill 的灵魂，agent 不要为了快而省掉提问直接写——那样就退化成 quickwrite，没必要
- 用户想法记录在"阅读思考/用户想法/"，本 skill 兼顾生产者和消费者：写新段时读旧想法，过程中用户冒新想法也要随手补记
- 一段写得不达意、用户来回改多次，是正常的——引导式就是允许反复改，别嫌烦就擅自替用户拍板
